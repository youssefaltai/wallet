# Linear Integration Rules

Linear is the single source of truth for all planned work. Every feature, bug, and audit finding lives in Linear. Every commit references an issue ID.

## Issue Lifecycle

```
Backlog → Todo → In Progress → In Review → Done
                                         ↘ Cancelled
```

- **Backlog**: discovered but not yet scheduled
- **Todo**: committed to the current cycle
- **In Progress**: actively being worked on — update to this state when you start
- **In Review**: PR open or awaiting validation
- **Done**: shipped and verified

Always update the issue state as work progresses. Do not leave issues "In Progress" after merging.

## PR Title & Commit Format

The **PR title** is the canonical place for the Linear issue ID — it becomes the squash merge commit on main:

```
feat: add recurring transaction detection [WALLET-42]
fix: prevent overdraft on concurrent transfers [WALLET-5]
chore: remove dead dependency [WALLET-18]
```

- Use the full identifier: `WALLET-XX` (not `WAL-XX`)
- Individual commits on a branch don't need the ID — only the PR title does
- If work spans multiple issues: `[WALLET-12, WALLET-15]`
- Pure chores with no Linear issue (dep bumps, typos): omit the tag

## Priority Mapping

| Severity (audit-guide.md) | Linear Priority |
|---------------------------|-----------------|
| CRITICAL (H1, H2)         | Urgent          |
| HIGH                      | High            |
| MEDIUM                    | Medium          |
| LOW                       | Low             |
| Feature request           | Medium          |
| Chore                     | Low             |

## Label Conventions

Apply labels when creating issues so they can be filtered:

| Label       | When to apply                                        |
|-------------|------------------------------------------------------|
| `bug`       | Something broken that worked before                  |
| `security`  | Auth, authz, injection, secrets exposure             |
| `ai`        | AI tools, system prompt, LLM integration             |
| `database`  | Schema, migrations, queries, indexes                 |
| `ui`        | React components, layouts, CSS, accessibility        |
| `performance` | N+1 queries, slow renders, bundle size             |
| `tech-debt` | Audit findings, refactors, test coverage             |
| `feature`   | New user-facing capability                           |

## When to Create Linear Issues

**Always create an issue for:**
- Audit findings (H/M/L severity)
- Bugs found in code review or testing
- New features (even small ones)
- Performance or security improvements

**Skip creating an issue for:**
- Typo fixes
- Chore commits under ~10 minutes (formatting, comment updates)
- Commits generated by `/evolve` on `.claude/` files

## Project and Cycle Conventions

- All Wallet work lives in the **Wallet** project
- Use cycles (sprints) to time-box work; don't leave issues in "In Progress" across cycles
- Milestones = major product milestones (beta, v1.0, etc.)
- Epics = large features spanning multiple issues

## Looking Up Issues

Before creating a new issue, check if it already exists:
```
list_issues → filter by title keywords
```

When referencing an issue in conversation, use `get_issue` to pull current state — don't rely on memory.

## Automation Touch Points

- `/triage` — Runs a live audit and creates Linear issues for all findings
- `/issue <desc>` — Creates a single issue and returns the ID
- `/feature` — Creates Linear issue → branch → implements → opens PR via `/ship`
- `/fix` — Finds/creates Linear issue → branch → fixes → opens PR via `/ship`
- `/ship` — Validates → pushes branch → `gh pr create` → sets issue to "In Review"
